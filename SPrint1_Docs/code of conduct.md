Our Shared Goals
Deliver a working full-stack app by the deadline specified by assessment brief.
Keep the codebase stable, readable, and reviewable.
Treat each other with respect and keep a calm and professional atmosphere.
Communication Rules
Primary channel: weekly meetings during the lab classes plus additional teams meeting agreed on in advance as needed.
Decisions live in writing: important choices (tech stack, database schema, routes, UI flow) must be posted in the channel or documented.
No sarcasm/insults/passive aggression. If something feels off, assume good intent first.
Not following these rules might result in consultation with a teacher and removing from  a group.
Meeting attendance
We meet once a week during the class labs with an agenda.
If you can’t attend, you message the group before the meeting.
Meetings end with:
what’s done
what’s next
who owns what
Being absent doesn’t remove the responsibility – meeting notes should be reviewed and tasks assigned completed.
Absence without a good reason, especially during the review, might result in a grade 0 for the sprint.
Tasks ownership
Every task has:
a clear owner (agreed during weekly meetings)
a “definition of done” 
a target date
Owners are responsible for moving the task forward and flagging blockers early.
If you take a task and go silent for 48 hours, the team can reassign it.



Conflict Resolution
Step 1: Discuss calmly.
Step 2: If still stuck, uninvolved member of group (if there is any) is supposed to fairly work as a mediator.
Step 3: If unresolved, the team votes or ask tutor for help
We do not keep arguing for hours.
Engineering Quality Standards
We don’t merge broken builds.
Basic expectations before merging:
app runs locally
no obvious console errors
feature tested manually (happy path + one edge case)
linting/formatting applied if used
Keep commits meaningful. 
 Git & branching rules
Main branch is always deployable.
No direct commits to main. Everything goes through a Pull Request (PR).
PR requirements:
short description + screenshots if UI
linked issue/task
at least 1 reviewer approval
Resolve merge conflicts with the owner + reviewer if needed.
Data & ethics
We do not store sensitive personal data unless required.
We don’t upload copyrighted content (full book scans, full album files).
Use realistic dummy data when demoing.



Academic integrity
We don’t plagiarize code and we cite any major external code/resources used.
If AI tools are used, we follow university rules and don’t submit generated code we don’t understand.
If someone is overwhelmed
Say it early. The team will rebalance tasks.
No shame — silence is the only thing that hurts the project.
